# Global Design Directive (GDD)
## Website Builder System - Universal Design Rulebook

**Version:** 1.0.0  
**Last Updated:** December 2024  
**Status:** Active Standard

---

## 1. Purpose of This Document

This Global Design Directive (GDD) defines the universal design system, component standards, layout rules, image rules, and UX principles for all websites generated by the Website Builder SaaS within the Website-Dev-Factory HQ framework.

**Core Principles:**
- **No Deviation Without Override**: All sites must adhere to these rules unless explicitly overridden through documented exceptions
- **Consistency First**: Every site should feel cohesive and professional, following the same foundational patterns
- **Scalability**: Rules are designed to work across industries while maintaining brand flexibility
- **Performance**: All standards prioritize Core Web Vitals and Lighthouse scores
- **Accessibility**: WCAG 2.1 AA compliance is mandatory, not optional

This document serves as the single source of truth for:
- Design system implementation
- Component architecture
- Image and media handling
- SEO and performance standards
- Code quality and accessibility requirements

**Document Authority:**
- All developers must reference this document before creating new components
- All design decisions must align with these standards
- Version changes require team review and approval
- Breaking changes require major version increments

## 2. Global Layout Philosophy

### 2.1 Spacing Scale

**Default Section Padding:**
- Mobile: `py-20` (5rem / 80px)
- Tablet: `py-24` (6rem / 96px)
- Desktop: `py-24` or `py-32` (8rem / 128px) for hero sections

**Container Padding:**
- Mobile: `px-4` (1rem / 16px)
- Tablet: `px-6` (1.5rem / 24px)
- Desktop: `px-6` or `px-8` (2rem / 32px)

**Element Spacing:**
- Between sections: `gap-16` to `gap-24` (4rem to 6rem)
- Within components: `gap-4` to `gap-8` (1rem to 2rem)
- Card internal padding: `p-6` to `p-8` (1.5rem to 2rem)

**Spacing Tokens:**
```css
--space-section-mobile: 5rem;    /* py-20 */
--space-section-desktop: 6rem;   /* py-24 */
--space-container-mobile: 1rem;  /* px-4 */
--space-container-desktop: 1.5rem; /* px-6 */
```

### 2.2 Grid Usage

**Standard Grid Patterns:**
- **Single Column**: Default for mobile, content-focused pages
- **Two Column**: `grid-cols-1 md:grid-cols-2` for content/image splits
- **Three Column**: `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` for service cards, features
- **Four Column**: `grid-cols-2 md:grid-cols-4` for logos, small cards (mobile-first)

**Grid Gaps:**
- Mobile: `gap-4` (1rem)
- Desktop: `gap-6` to `gap-8` (1.5rem to 2rem)

**Container Max Widths:**
- Content: `max-w-4xl` (56rem / 896px)
- Wide Content: `max-w-6xl` (72rem / 1152px)
- Full Width: `max-w-7xl` (80rem / 1280px)

### 2.3 Breakpoints (Mobile-First)

**Standard Tailwind Breakpoints:**
- **Mobile**: Default (0px - 767px)
- **Tablet**: `md:` (768px+)
- **Desktop**: `lg:` (1024px+)
- **Large Desktop**: `xl:` (1280px+)
- **Extra Large**: `2xl:` (1536px+)

**Usage Pattern:**
```html
<!-- Always mobile-first, then scale up -->
<div class="text-base md:text-lg lg:text-xl">
  Content scales up
</div>
```

**Responsive Design Rules:**
- Design for mobile first, enhance for larger screens
- Never hide critical content on mobile
- Touch targets minimum 44x44px on mobile
- Navigation must be fully accessible on all screen sizes

### 2.4 Section Structure Patterns

**Standard Section Template:**
```html
<section class="py-20 md:py-24 bg-white">
  <div class="container mx-auto px-4 md:px-6 max-w-7xl">
    <!-- Section Header (Optional) -->
    <div class="text-center mb-12 md:mb-16">
      <h2 class="text-3xl md:text-4xl lg:text-5xl font-bold mb-4">
        Section Title
      </h2>
      <p class="text-lg md:text-xl text-gray-600 max-w-3xl mx-auto">
        Section description
      </p>
    </div>
    
    <!-- Section Content -->
    <div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6 md:gap-8">
      <!-- Content items -->
    </div>
  </div>
</section>
```

**Section Types:**
1. **Hero Sections**: Full viewport height, centered content, background image
2. **Content Sections**: Standard padding, container width, readable line length
3. **Feature Sections**: Grid-based, icon + text patterns
4. **Testimonial Sections**: Carousel or grid, quote + attribution
5. **CTA Sections**: Centered, prominent, action-focused

### 2.5 Typography Hierarchy

**Font System:**
- **Primary Font**: System sans-serif stack (DM Sans, Inter, or similar)
- **Display Font**: Distinctive heading font (Outfit, Poppins, or similar)
- **Monospace**: Code/technical content only

**Heading Scale:**
- **H1**: `text-4xl md:text-5xl lg:text-6xl xl:text-7xl` (Hero only, one per page)
- **H2**: `text-3xl md:text-4xl lg:text-5xl` (Section titles)
- **H3**: `text-2xl md:text-3xl` (Subsection titles)
- **H4**: `text-xl md:text-2xl` (Card titles, feature titles)
- **H5**: `text-lg md:text-xl` (Minor headings)
- **H6**: `text-base md:text-lg` (Labels, captions)

**Body Text:**
- **Base**: `text-base` (16px)
- **Large**: `text-lg` (18px) for lead paragraphs
- **Small**: `text-sm` (14px) for captions, metadata
- **Line Height**: `leading-relaxed` (1.625) for body, `leading-tight` (1.25) for headings

**Font Weight Scale:**
- **Light**: `font-light` (300) - Rare, decorative only
- **Normal**: `font-normal` (400) - Body text default
- **Medium**: `font-medium` (500) - Emphasis, buttons
- **Semibold**: `font-semibold` (600) - Subheadings
- **Bold**: `font-bold` (700) - Headings, strong emphasis

### 2.6 Hero Structure Patterns

**Primary Hero Pattern:**
```html
<section class="relative min-h-[90vh] flex items-center justify-center overflow-hidden">
  <!-- Background Image -->
  <div class="absolute inset-0 z-0">
    <img src="hero-image.webp" alt="..." class="w-full h-full object-cover opacity-90" />
    <!-- Overlay for text readability -->
    <div class="absolute inset-0 bg-gradient-to-t from-black/80 via-black/40 to-transparent"></div>
  </div>
  
  <!-- Content Container -->
  <div class="relative z-10 container mx-auto px-4 md:px-6 text-center text-white">
    <h1 class="text-4xl md:text-6xl lg:text-7xl font-bold mb-6">
      Main Headline
    </h1>
    <p class="text-xl md:text-2xl mb-8 max-w-3xl mx-auto">
      Subheadline text
    </p>
    <div class="flex flex-col sm:flex-row gap-4 justify-center">
      <a href="/contact" class="btn-primary">Primary CTA</a>
      <a href="/services" class="btn-secondary">Secondary CTA</a>
    </div>
  </div>
</section>
```

**Secondary Hero Pattern:**
- Reduced height: `min-h-[60vh]`
- Content-left or content-right alignment options
- Image + text side-by-side on desktop
- Stacked on mobile

**Hero Requirements:**
- One H1 per hero (never duplicate)
- Clear value proposition in headline
- Single primary CTA, optional secondary
- Trust indicators (badges, ratings) above fold
- Background image must have overlay for text contrast

## 3. Component Library Standards

### 3.1 Header & Mobile Menu

**Desktop Header Structure:**
```html
<header class="fixed top-0 z-50 w-full bg-white/90 backdrop-blur-md border-b border-gray-200">
  <div class="container mx-auto px-4 md:px-6 flex items-center justify-between h-20">
    <!-- Logo -->
    <a href="/" class="flex-shrink-0">
      <img src="/logo.png" alt="Brand Name" class="h-10 md:h-12" />
    </a>
    
    <!-- Desktop Navigation -->
    <nav class="hidden md:flex items-center gap-1">
      <a href="/services" class="px-4 py-2 rounded-full hover:bg-gray-100">Services</a>
      <!-- More nav items -->
    </nav>
    
    <!-- CTA Button -->
    <a href="/contact" class="hidden md:inline-flex btn-primary">Get Quote</a>
    
    <!-- Mobile Menu Toggle -->
    <button class="md:hidden" aria-label="Open menu">☰</button>
  </div>
</header>
```

**Mobile Menu Requirements:**
- **Z-Index**: `z-[60]` (above header `z-50`)
- **Animation**: Slide-in from right, `translate-x-full` to `translate-x-0`
- **Backdrop**: Dark overlay with `backdrop-blur-sm`
- **Close Button**: Visible X button in top-right
- **Menu Items**: Full-width, touch-friendly (min 48px height)
- **Auto-Close**: Close on link click, backdrop click, or ESC key

**Header Behavior:**
- **Sticky**: Fixed position, stays on scroll
- **Transparent Variant**: `bg-transparent` on hero, transitions to `bg-white/90` on scroll
- **Scroll State**: Use `data-scrolled` attribute for state management
- **Logo Sizing**: Responsive, smaller on scroll (optional)

**Accessibility:**
- `aria-label` on menu toggle
- `aria-expanded` on menu button
- `aria-hidden` on menu when closed
- Keyboard navigation support
- Focus trap when menu open

### 3.2 Hero Sections

**Primary Hero Variant:**
- **Height**: `min-h-[90vh]` (full viewport)
- **Content**: Centered, single column
- **Background**: Full-bleed image with overlay
- **Elements**: H1, subheadline, 1-2 CTAs, optional trust badge
- **Spacing**: `gap-8` between elements

**Secondary Hero Variant:**
- **Height**: `min-h-[60vh]`
- **Layout**: Two-column on desktop (image + content)
- **Alignment**: Left or right content alignment
- **Background**: Optional image, can use gradient
- **Elements**: H2 (not H1), description, single CTA

**Trust Badge Pattern:**
```html
<div class="inline-flex items-center gap-2 rounded-full border border-white/20 bg-white/10 px-4 py-1.5 backdrop-blur-md">
  <span class="relative flex h-2 w-2">
    <span class="absolute animate-ping rounded-full bg-green-400 opacity-75"></span>
    <span class="relative rounded-full bg-green-500"></span>
  </span>
  <span class="text-sm font-medium text-white">Trust Indicator Text</span>
</div>
```

**Social Proof Pattern:**
- 4-6 circular avatars (overlapping, `-space-x-3`)
- Star rating (5 stars)
- Trust text ("Trusted by X customers")
- Rounded container with backdrop blur

### 3.3 CTA Blocks

**Standard CTA Structure:**
```html
<section class="py-20 md:py-24 bg-primary-600 text-white">
  <div class="container mx-auto px-4 md:px-6 text-center max-w-4xl">
    <h2 class="text-3xl md:text-4xl font-bold mb-4">Ready to Get Started?</h2>
    <p class="text-xl mb-8 opacity-90">Compelling call-to-action copy</p>
    <div class="flex flex-col sm:flex-row gap-4 justify-center">
      <a href="/contact" class="btn-primary-white">Primary Action</a>
      <a href="/services" class="btn-secondary-white">Learn More</a>
    </div>
  </div>
</section>
```

**CTA Variants:**
1. **Full-Width**: Background color, centered content
2. **Card-Based**: White card on colored background
3. **Inline**: Within content sections, smaller scale
4. **Sticky**: Fixed bottom on mobile (optional)

**CTA Requirements:**
- Clear, action-oriented copy
- Single primary action
- Optional secondary action (never more than 2)
- Mobile-friendly button sizing
- Accessible contrast ratios

### 3.4 Testimonial Sections

**Testimonial Grid Pattern:**
```html
<section class="py-20 md:py-24 bg-gray-50">
  <div class="container mx-auto px-4 md:px-6">
    <h2 class="text-3xl md:text-4xl font-bold text-center mb-12">What Our Customers Say</h2>
    <div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6 md:gap-8">
      <!-- Testimonial Card -->
      <div class="bg-white rounded-2xl p-6 md:p-8 shadow-lg">
        <div class="flex items-center gap-1 mb-4 text-yellow-400">
          <!-- 5 stars -->
        </div>
        <blockquote class="text-gray-700 mb-4">"Testimonial quote text"</blockquote>
        <div class="flex items-center gap-3">
          <img src="avatar.webp" alt="Customer name" class="w-12 h-12 rounded-full" />
          <div>
            <p class="font-semibold">Customer Name</p>
            <p class="text-sm text-gray-500">Location, Service</p>
          </div>
        </div>
      </div>
    </div>
  </div>
</section>
```

**Testimonial Requirements:**
- **Minimum**: 3 testimonials per section
- **Maximum**: 6 testimonials in grid (3 columns max)
- **Elements**: Quote, star rating, customer name, location/service, avatar
- **Card Style**: White background, rounded corners, shadow
- **Accessibility**: Proper `blockquote` usage, alt text for avatars

**Steps Pattern:**

**Steps Requirements:**
- **Count**: 3-5 steps (optimal: 4)
- **Numbering**: Visual number indicator (circle with number)

- **Spacing**: Equal spacing between steps
- **Icons**: Optional icon above number

- **Copyright**: Current year, centered bottom bar

## 4. Imagery Rules

### 4.1 When to Use AI-Generated Images

**Use AI-Generated Images For:**
- **Hero Backgrounds**: High-quality, brand-aligned hero images
- **Service-Specific Images**: Images that match exact service descriptions
- **Location-Specific Images**: Images that represent specific geographic areas
- **Testimonial Avatars**: When real customer photos unavailable
- **Feature Illustrations**: Abstract concepts, processes, benefits

**AI Image Requirements:**
- **Style Consistency**: All AI images must match brand aesthetic
- **Quality**: Minimum 1200px width, high resolution
- **Relevance**: Must directly relate to content
- **Diversity**: Represent diverse people when including humans
- **Brand Alignment**: Colors, mood, style must match brand

**AI Image Prompts:**
- Include brand colors in prompts
- Specify aspect ratios (16:9 for heroes, 4:3 for cards, 1:1 for avatars)
- Request specific lighting, mood, composition
- Avoid generic stock photo aesthetics

### 4.2 When to Use Stock Images

**Use Stock Images For:**
- **Generic Backgrounds**: When specific location/service combo not needed
- **Icon Replacements**: Simple illustrations, patterns
- **Placeholder Content**: During development, before final assets ready
- **Blog/Content Images**: Supporting images for articles

**Stock Image Requirements:**
- **Licensing**: Only use properly licensed stock images
- **Quality**: High resolution, professional quality
- **Brand Fit**: Must align with brand aesthetic
- **Uniqueness**: Avoid overused stock photos
- **Attribution**: Include if required by license

### 4.3 Never Use Brand Logo as Hero Image

**Critical Rule:**
- **NEVER** use the company logo as a hero background image
- **NEVER** use the logo as a service card image
- **NEVER** use the logo as a location image

**Logo Usage:**
- **Header**: Logo in navigation only
- **Footer**: Logo in footer only
- **Favicon**: Logo as favicon/icon only
- **Social Sharing**: Logo in OG images (small, corner placement)

**Hero Image Alternatives:**
- Real service photos (lawn, landscaping work)
- Location-specific imagery (neighborhoods, properties)
- AI-generated brand-aligned images
- Stock photos that match service/location

### 4.4 Image Aspect Ratio Expectations

**Standard Aspect Ratios:**
- **Hero Images**: 16:9 (1920x1080px minimum)
- **Service Cards**: 4:3 (1200x900px) or 1:1 (1200x1200px)
- **Location Cards**: 4:3 (1200x900px)
- **Testimonial Avatars**: 1:1 (200x200px minimum)
- **Feature Icons**: 1:1 (varies by size)
- **Blog Images**: 16:9 or 4:3

**Responsive Image Sizing:**
```html
<picture>
  <source srcset="image-1200.webp" media="(min-width: 1024px)" />
  <source srcset="image-800.webp" media="(min-width: 768px)" />
  <img src="image-600.webp" alt="..." class="w-full h-auto" />
</picture>
```

**Image Dimensions:**
- **Mobile**: 600px width
- **Tablet**: 800px width
- **Desktop**: 1200px width
- **Large Desktop**: 1920px width (hero only)

### 4.5 Image Optimization (WebP Requirement)

**Format Requirements:**
- **Primary Format**: WebP for all images
- **Fallback**: JPG or PNG for older browsers
- **SVG**: For logos, icons, simple graphics

**WebP Implementation:**
```html
<picture>
  <source srcset="image.webp" type="image/webp" />
  <img src="image.jpg" alt="Description" />
</picture>
```

**Optimization Standards:**
- **Compression**: 80-85% quality for photos
- **File Size**: Hero images < 300KB, cards < 100KB
- **Lazy Loading**: All images below fold use `loading="lazy"`
- **Priority**: Hero images use `loading="eager"` or `fetchpriority="high"`

**Image Processing:**
- Generate multiple sizes (600, 800, 1200, 1920px)
- Convert to WebP with fallback
- Optimize file names (descriptive, SEO-friendly)
- Include proper alt text in filenames for reference

### 4.6 Placement Rules

**Hero Images:**
- Full-bleed background, `object-cover` to fill container
- Overlay for text readability (gradient or darkening)
- Never place text directly on complex backgrounds without overlay

**Card Images:**
- Top of card, full width
- Aspect ratio maintained, `object-cover` for consistent heights
- Optional gradient overlay for text contrast

**Content Images:**
- Inline with text, responsive width
- Caption below image when needed
- Proper spacing from text (margin top/bottom)

**Avatar Images:**
- Circular (`rounded-full`)
- Consistent sizing within same component
- Border for contrast if needed

### 4.7 Fallback Rules if Images Fail to Load

**Fallback Strategy:**
```html
<img 
  src="image.webp" 
  alt="Description"
  onerror="this.onerror=null; this.src='fallback.jpg';"
  class="bg-gray-200"
/>
```

**Fallback Requirements:**
- **Background Color**: Light gray background (`bg-gray-200`) while loading
- **Alt Text**: Always include descriptive alt text
- **Error Handling**: Graceful degradation to fallback image
- **Loading State**: Skeleton loader or placeholder during load
- **Missing Image**: Show placeholder icon + text if image completely unavailable

**Placeholder Patterns:**
- **Service Cards**: Gradient background matching brand colors
- **Avatars**: Initials in circle, or generic user icon
- **Hero**: Solid color or subtle gradient (never broken image icon)

## 5. Brand & Design System Rules

### 5.1 Tailwind-Only Styling

**Core Principle:**
- **No Custom CSS**: All styling must use Tailwind utility classes
- **No Inline Styles**: Except for dynamic values (colors from CMS)
- **No Style Tags**: Component-level styles only when absolutely necessary
- **CSS Variables**: Use for design tokens (colors, spacing) only

**Tailwind Configuration:**
- Extend theme in `tailwind.config.js` for brand colors
- Use design tokens for consistent spacing, typography
- Custom utilities only for repeated patterns

**Allowed Exceptions:**
- CSS animations that can't be achieved with Tailwind
- Third-party component library overrides (minimal)
- Print styles (if needed)

### 5.2 Font System

**Font Stack:**
```css
--font-sans: "DM Sans", system-ui, -apple-system, sans-serif;
--font-display: "Outfit", system-ui, -apple-system, sans-serif;
--font-mono: "Fira Code", "Courier New", monospace;
```

**Font Loading:**
- Preload critical fonts
- Use `font-display: swap` for web fonts
- Fallback to system fonts during load
- Limit to 2 font families maximum

**Font Usage:**
- **Body Text**: Sans-serif stack
- **Headings**: Display font (optional, can use sans-serif)
- **Monospace**: Code blocks, technical content only

### 5.3 Color System

**Color Selection:**
- **Primary Color**: Brand's main color (blue, green, etc.)
- **Secondary Color**: Accent color for CTAs, highlights
- **Neutral Colors**: Gray scale for text, backgrounds
- **Semantic Colors**: Success (green), error (red), warning (yellow)

**Color Scale:**
- Use 50-950 scale (50 lightest, 950 darkest)
- Primary: `primary-500` (base), `primary-600` (hover), `primary-400` (light)
- Neutral: `gray-50` to `gray-950` for backgrounds, `gray-600` to `gray-900` for text

**Arbitrary Values:**
- **Avoid**: Use design tokens instead
- **Allowed**: Only for one-off brand colors not in system
- **Document**: Any arbitrary values must be documented

**Color Accessibility:**
- **Contrast**: Minimum 4.5:1 for body text, 3:1 for large text
- **Focus States**: High contrast focus rings
- **Error States**: Red for errors, never rely on color alone

### 5.4 Button Styling

**Minimum Required Classes:**
```html
<!-- Primary Button -->
<a href="#" class="inline-flex items-center justify-center px-6 py-3 bg-primary-600 text-white font-medium rounded-lg hover:bg-primary-700 focus:outline-none focus:ring-2 focus:ring-primary-500 focus:ring-offset-2 transition-colors">
  Button Text
```

- **Location-Service Pages**: Link to base service page, location page, related services

**Anchor Text Rules:**
- **Descriptive**: Use descriptive anchor text, not "click here"
- **Keywords**: Include relevant keywords naturally
- **Context**: Link text should make sense out of context

**Linking Density:**
- **Homepage**: 10-15 internal links
- **Service Pages**: 5-10 internal links
- **Location Pages**: 8-12 internal links
- **Location-Service Pages**: 5-8 internal links

### 8.6 URL Structure Rules

**URL Requirements:**
- **Lowercase**: All URLs lowercase
- **Hyphens**: Use hyphens, not underscores
- **Descriptive**: Include keywords in URL
- **Short**: Keep URLs concise (3-5 words max)
- **No Parameters**: Avoid query parameters for content pages

**URL Examples:**
- ✅ Good: `/services/lawn-care`
- ✅ Good: `/locations/ocean-view/lawn-care`
- ❌ Bad: `/services/lawn_care`

### 8.7 301 Redirect Mapping Template

**Redirect Rules:**
- **Old URLs**: Redirect to new URL structure
- **Trailing Slashes**: Consistent (prefer no trailing slash)
- **WWW**: Redirect www to non-www (or vice versa, be consistent)
- **HTTPS**: Force HTTPS, redirect HTTP

**Redirect Template:**
```
# Old URL → New URL
/services/old-service → /services/new-service (301)
/location/old-name → /locations/new-name (301)
/page.html → /page (301)
```

**Implementation:**
- Use server-level redirects (Apache `.htaccess`, Nginx config)
- Or use framework redirects (Astro `redirects`, Next.js `redirects`)
- Test all redirects after implementation

## 11. Versioning Rules

### 11.1 Version Numbering

**Semantic Versioning:**
- **Major Version** (1.0.0): Breaking changes, incompatible updates
- **Minor Version** (1.1.0): New features, backward-compatible
- **Patch Version** (1.0.1): Bug fixes, minor updates

**Version Format:**
```
v{major}.{minor}.{patch}
Example: v1.2.3
```

### 11.2 Backward Compatibility

**Backward Compatibility Rules:**
- **Minor/Patch Updates**: Must remain backward-compatible
- **Component APIs**: Existing props must continue to work
- **Class Names**: Tailwind classes remain stable
- **URL Structure**: Existing URLs must not break
- **Data Structure**: JSON schemas remain compatible

**Breaking Changes:**
- **Major Version Required**: Any breaking change requires major version bump
- **Migration Guide**: Provide migration guide for major updates
- **Deprecation Period**: Deprecate features before removing (one version cycle)

### 11.3 Update Process

**Version Update Workflow:**
1. **Review Changes**: Document all changes in changelog
2. **Determine Version**: Assess if changes are major, minor, or patch
3. **Update Document**: Update version number and date in GDD
4. **Update Changelog**: Document changes in changelog section
5. **Team Review**: Get team approval for major version changes
6. **Communication**: Notify team of updates, especially breaking changes

**Changelog Format:**
```markdown
## [Version] - YYYY-MM-DD

### Added
- New feature or component

### Changed
- Modified existing feature

### Fixed
- Bug fix

### Removed
- Deprecated feature (major version only)
```

### 11.4 Document Maintenance

**Update Frequency:**
- **As Needed**: Update when standards change
- **Quarterly Review**: Review document quarterly for accuracy
- **Major Updates**: Review before major framework changes

**Change Approval:**
- **Minor/Patch**: Developer can update with documentation
- **Major**: Requires team review and approval
- **Breaking Changes**: Requires stakeholder approval

**Documentation Standards:**
- **Clarity**: All rules must be clear and unambiguous
- **Examples**: Include code examples where helpful
- **Rationale**: Explain "why" for important rules
- **Exceptions**: Document any allowed exceptions
